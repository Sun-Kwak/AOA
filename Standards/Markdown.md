# Markdown

## Purpose

AIOS의 모든 문서를 작성할 때의 마크다운 규칙을 정의한다.
일관된 형식이 있어야 AI가 문서를 정확하게 파싱하고
사람이 읽기에도 일관성이 유지된다.

---

## Scope

AIOS 내 모든 `.md` 파일에 적용된다.
코드 파일, YAML 파일은 해당하지 않는다.

---

## 기본 구조 규칙

### 제목 계층

```
# H1   — 파일당 하나. 문서 제목.
## H2  — 주요 섹션
### H3 — 서브 섹션
#### H4 — 필요한 경우에만 (가급적 피함)
```

H1은 파일당 하나만 사용한다.
H4 이상 깊이는 가급적 사용하지 않는다.

---

### 섹션 구분

섹션 사이는 `---`로 구분한다.

```markdown
## 섹션 A

내용

---

## 섹션 B

내용
```

---

### 필수 섹션 (Core / Policy / Standard 문서)

Core, Policies, Standards 문서는 반드시 포함:

```markdown
# 문서 제목

## Purpose
이 문서의 역할 (1~3 문장)

---

## Scope
무엇을 다루고 무엇을 다루지 않는지

---

## (본문 섹션들)

---

## TODO
미구현 항목
```

---

## 테이블 규칙

- 테이블은 반드시 헤더 행 포함.
- 헤더와 데이터 사이 구분선 포함.
- 열 너비는 정렬 불필요 (렌더링 시 자동 처리).

```markdown
| 컬럼 A | 컬럼 B | 컬럼 C |
|--------|--------|--------|
| 값 1   | 값 2   | 값 3   |
```

---

## 코드 블록 규칙

- 모든 코드 블록에 언어를 명시한다.
- 언어 불명확 시 `text` 사용.
- 인라인 코드는 파일명, ID, 경로, 명령어에 사용.

```markdown
```yaml
key: value
```

```text
일반 텍스트 예시
```

파일명: `manifest.yaml`
경로: `Projects/<name>/`
```

---

## 목록 규칙

- 순서가 있는 절차 → 번호 목록
- 순서가 없는 항목 → 글머리 목록
- 중첩은 2단계까지만

```markdown
1. 첫 번째 단계
2. 두 번째 단계
   - 세부 항목
   - 세부 항목
3. 세 번째 단계
```

---

## 강조 규칙

| 형식 | 용도 |
|------|------|
| `**굵게**` | 중요한 규칙, 경고 |
| `` `코드` `` | 파일명, ID, 경로, 명령어 |
| `>` 인용 | 참고 사항, 부가 설명 |

이탤릭(`*기울임*`)은 AIOS 문서에서 사용하지 않는다.

---

## 체크리스트 규칙

완료 조건, 검증 항목에는 체크리스트를 사용한다:

```markdown
- [ ] 미완료 항목
- [x] 완료 항목
```

---

## AI 파싱을 위한 규칙

AI가 문서를 읽을 때 혼동을 줄이기 위해:

- 섹션 제목은 행동(동사)이나 명사로 명확히.
- 하나의 섹션에 하나의 주제만.
- 목록 항목은 짧고 명확하게 (한 항목에 여러 개념 금지).
- 예시 코드는 실제 동작 가능한 형태로.
